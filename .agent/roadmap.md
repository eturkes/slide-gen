# slide-gen — roadmap

Goal: a CLI + home for slides across `~/Projects/*`. Discover sibling projects → toggle on/off → `codex exec` generates a `deck.html`-style deck per enabled project → render to PDF. The repo is the deck home: `templates/deck.html` = style contract; `decks/<project>/` = generated output.

Stack: **MoonBit (native backend)** — the agent-oriented bet (AGENTS.md ethos). UX = flag subcommands (no TUI). State = gitignored local. Discovery = all sibling dirs, default-off. (Owner-confirmed at M2 plan.)

Seed: commit `0526d53` (repo init: deck.html, agent guidance, `.agent` boilerplate) + the `templates/deck.html` move. `deck.html` = the content-stripped generalization of the proven prototype `~/Projects/spine-segmentation/docs/slides/` (filled deck.html + figures.py + render.py → PDF) — M2/M3 mirror it (mechanics in memory.md).

## M2 — Generation (REVIEWED)

As-designed: `slide-gen generate [<project>]` drives `codex exec` (one shot per project) to author a deck BUNDLE at `decks/<project>/` — `deck.html` (template-styled) + `provenance.json` (per-claim source citations) + `gaps.md` (unverifiable / human-needed) + `figures.py` + `assets/*.png`. A deterministic MoonBit VALIDATOR gates every bundle (template conformance + provenance quote-grep + furigana structural guards); a project-local Python analyzer cross-checks furigana readings (hybrid). Deck CONTENT = codex's job (nondeterministic, env-gated + external `timeout`); the harness + validator are the testable MoonBit surface.

### Decisions (owner + research, at plan; env/tool mechanics → memory.md)
- **Gate PASSED** — `codex exec` with an explicit `--sandbox` IGNORES the interactive trust gate (probe-verified: read-only exec on a non-trusted sibling → exit 0, replied); `trust_level` only gates project-local `.codex/` loading, not runtime.
- **Sandbox** = `--dangerously-bypass-approvals-and-sandbox` (owner; the container is externally sandboxed). Harness must still direct writes to the bundle dir (bypass = no write confinement).
- **Input scope** = codex may read ANYTHING (owner; no deny-list / no input curation). The provenance **quote-grep** anti-hallucination check stays independent of that.
- **Furigana** = HYBRID — codex writes the three tiers incl. furigana → MoonBit structural guards (`.jp`==rt-stripped-`.jpf` · `<rt>` all-kana · ruby covers kanji runs) → Python analyzer (fugashi+UniDic / SudachiPy) independently re-reads the JA; disagreements / analyzer-unknowns → `gaps.md`.
- **Output** = COMMITTED deck home — track `decks/<project>/{deck.html,provenance.json,gaps.md,figures.py,assets/*.png}`; gitignore only render intermediates (`_deck_full.png`, `page_*.png`). **Privacy**: the no-secrets/PHI rule covers the WHOLE committed bundle (quote-grep COPIES verbatim source text into `provenance.json`, so a cited secret would leak). No deterministic secret gate (owner chose read-anything + committed) → prompt guideline + human review before the owner pushes (owner handles remote).
- **Provenance contract**: `provenance.json` rows `{id,slide,claim,status:verified|inferred|gap,src:"path:Lstart-Lend" (parent-relative or absolute),quote:"<verbatim>"}`; deck claims carry `data-claim="id"`. The quote must LITERALLY occur WITHIN the cited `src` line-slice (a cheap EXISTENCE check — kills invented quotes; NOT a truth check — misattribution / cherry-pick still pass → analyzer + human). A row must NOT cite a generated bundle file (no self-citation). Prompt drives a codex DRAFT→re-verify-quotes+line-numbers→FINAL self-critic.
- **Section-count gotcha**: count `<section class="slide">` after COMMENT-STRIP — the template's line-8 header comment inflates a raw `grep -c` by 1.
- **Figures**: codex authors `figures.py` from COMMITTED, CITED numbers and runs it (bypass permits exec) → `assets/*.png`; raw-data or failed figures → a `.fig-ph` placeholder + a `gaps.md` entry (human runs it, à la the prototype's `figure_example.py`). Each committed `assets/*.png` earns a `provenance.json` row citing the real number-SOURCES it plots (NEVER `figures.py` — a bundle file is self-citation). figures.py + assets are CONDITIONAL: a text-only deck has neither → `validate_bundle_dir` requires only deck.html + provenance.json + gaps.md, checking figures.py/assets solely when the deck computes figures.

## M3 — Rendering (REVIEWED)

Scope: `slide-gen render <project>` consumes one committed `decks/<project>/deck.html` + its committed PNG assets and publishes derived, gitignored `renders/<project>/{page_NN.png,deck.pdf}`. Render is source-project-independent (a committed deck remains renderable after its sibling disappears), never reruns `figures.py`, and has no enabled-set batch mode; M4 owns generate→render `run`, enabled orchestration, distribution, and docs.

### Decisions + verified gate

- **Gate PASSED** — clean worktree; `just all` = 114 MoonBit + 7 Python tests, 0-warn/lint. Runtime surface present: `chromiumfish path`, ChromiumFish browser, Noto Sans/CJK JP, GNU `timeout`, `uv`, and `img2pdf`; no BLOCKED units.
- **Raster screen path, DSF=1** — explicit `--force-device-scale-factor=1` + `--window-size=2560,1440` preserves the template's screen-CSS contract and emits exact 2560×1440 RGB PNGs. Chromium vector print is rejected for M3: the local proof converted the CSS 2560×1440 page to 1920×1080 PDF points (CSS 96 px/in vs PDF 72 pt/in), carried timestamps, and bypassed the required PNG-first artifact path. Higher DSF multiplies storage without adding source detail. Raster bytes are reproducible only on the same browser/font surface; the live gate reruns and compares there.
- **One viewport per page** — avoid the prototype's unbounded tall bitmap and any crop dependency. Chromium first `--dump-dom`s the validated stage-local deck snapshot; the shared context-aware scanner counts actual `section.slide` DOM elements (comment literals stay inert). For page `i`, a tiny stage-local iframe wrapper uses height `count×1440` and top `-i×1440`; a 2560×1440 viewport captures `page_NN.png`. No fragments, virtual-time budget, or compositor-drain flags. External `timeout --kill-after` bounds tool resolution plus every browser/PDF process; Chromium's noisy stderr is advisory when exit=0 and the exact artifact validates.
- **Offline/static render boundary** — strengthen the deterministic validator + authoring prompt before browser use: deck HTML is inert; executable/embedded/navigating elements, event handlers, non-asset resource URLs, and CSS imports/URLs are blocking. Chromium gets an isolated stage-local profile plus a dead proxy, loopback-bypass subtraction, catch-all failed DNS, and background-network suppression. Render reads only committed deck files; generated wrappers/profile/output stay under `renders/`. This is defense-in-depth around mandatory `--no-sandbox`, not a claim that Chromium is a security boundary.
- **Lossless deterministic PDF container** — uv-lock current `img2pdf==0.6.3`; invoke `--nodate --engine=internal --imgsize=72dpi --rotation=none` over ordered PNG argv. Local proof: repeated page capture hashes matched; repeated two-page PDF hashes matched; `pdfinfo` reported two 2560×1440-point pages and `pdfimages` reported the original 2560×1440 RGB streams at 72 dpi. Thus 1 px = 1 PDF point, matching the prototype.
- **Output isolation/publication** — derived artifacts live outside M2's closed committed bundle. Build in reserved `renders/.partial/<project>`, publish to `renders/<project>` through the existing no-follow backup/rollback/recovery protocol (`renders/.previous/<project>`), and keep the prior complete render on render failure. A closed staged-render layout prevents wrappers/profile/stale pages from publishing. Ignore all `renders/` artifacts. Generation/render entries share a non-blocking per-project advisory lease under `renders/.locks/`; successful regeneration invalidates that project's old render before deck promotion, so an older render cannot publish afterward. Losing a derived artifact if the later deck rename fails is preferable to retaining a PDF known to be stale.

## M4 — Integration / packaging / docs (IN-PROGRESS)

Scope: `slide-gen run [<project>]` is the production generate→render path. A name runs one current sibling even when disabled; omission runs the enabled set sequentially. Each successful item leaves a validator-clean committed deck plus `renders/<project>/{page_NN.png,deck.pdf}`. M4 also supplies a Debian/Linux checkout-bound install, a human README, deterministic CI, and one explicit-spend live proof. It does not publish a standalone binary/archive: the mutable Git checkout is intentionally the deck home and carries runtime templates, Python sidecars/lock, committed bundles, and local state.

### Decisions + verified gate

- **Gate PASSED** — clean worktree; `just all` = 157 MoonBit + 7 Python tests, 0-warn/lint; `just render-probe` = live DOM/raster/PDF probes green; `just live-render` = both committed six-page decks repeat byte-identically. Runtime/auth surface present: Codex logged in, ChromiumFish browser, Noto Sans CJK JP, GNU `timeout`, uv, Git, Poppler, and just. Current MoonBit v0.10.4 local-install probe emitted a working native ELF; no BLOCKED units.
- **One lease per end-to-end item** — refactor generation/render into package-owned already-locked seams while preserving their public lock-owning entries. `run` acquires the existing non-blocking per-project advisory lease once, generates, and renders before release; it never deadlocks by nesting the public entries. Any generation failure skips render. A failure before render invalidation preserves the prior pair; once a final-valid stage invalidates the old render, a later deck-rename failure may leave the old deck without its derived render, matching M3's accepted stale-safety boundary. If deck promotion succeeds but the new render fails, retain the valuable new deck and report a phase-specific partial failure with no stale PDF rather than rolling it back merely to make the pair look successful.
- **Batch semantics** — named `run` requires a current discovery candidate; no-arg `run` uses the persisted enabled set, preserves discovery order, appends sorted stale-state failures, continues after each project failure, and exits 1 iff any item failed. Empty enabled set is a successful zero-item summary. `run` has no `--dry-run`: a render of not-yet-generated content cannot be previewed honestly; `generate --dry-run` remains the exact Codex-spend preview.
- **Checkout-bound native install** — MoonBit's 2026-07 `moon install [SOURCE]` now installs native main packages and accepts local paths/`--bin`; the probe of `./cmd/main` installed a binary literally named `main`. Rename the executable package to `cmd/slide-gen`, stage it repo-locally with `moon install`, and install a thin POSIX launcher symlink into a configurable user bin (default `~/.local/bin`). The launcher resolves its own symlink back to this checkout, enters the repo, and execs the staged native binary, so every command works from unrelated directories without duplicating or guessing the deck home. Installation is idempotent but fails rather than replacing an unrelated destination. Mooncakes/Git-URL install is deliberately unsupported because it installs only the executable, not the required mutable app/resource home.
- **Quality split** — keep `just all` as the local format-in-place sweep; add a read-only `just ci` using `moon fmt --check`, frozen dependencies, denied MoonBit warnings, locked uv, shell lint, tests, release/install smoke, and a tracked before/after fingerprint (so intended uncommitted work is allowed but gate-induced mutation is not). GitHub Actions runs only this deterministic gate on pinned Ubuntu/MoonBit/uv/action revisions with read-only repository permission; Codex and Chromium gates stay explicit local jobs because they require auth, spend, and the canonical browser/font surface.
- **Acceptance boundary** — an opt-in `just live-run` builds a disposable one-project sibling/repo fixture, invokes real Codex then real Chromium/PDF through the production `run` path, validates the promoted bundle/render, inspects the PDF structure plus rasterized pages, and proves the tracked source tree unchanged. The install smoke separately invokes the installed launcher from outside the checkout. README must surface the whole-bundle privacy review, bypassed-Codex residuals, provenance limits, external prerequisites, state/output layout, failure semantics, and every deterministic/live gate.
